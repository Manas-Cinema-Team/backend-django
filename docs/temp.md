1. Закрыть auth до конца и зафиксировать контракт.
    - Добавить refresh endpoint.
    - Принять одно решение по URL: либо /api/auth/*, либо /api/v1/auth/*, и больше не менять.
    - Проверить зависимости и тестовый запуск на Postgres.
2. Спроектировать и завести доменные модели без API-логики.
    - Movie, Hall, MovieSession/Session, TicketPrice, SeatHold, Booking, BookingSeat.
    - Сразу заложить choices, индексы и уникальные ограничения.
    - BookingSeat и SeatHold лучше делать сразу, не “потом”, иначе будет переделка бронирования.
3. Сразу после моделей поднять admin.
    - По ТЗ фильмы, залы, сеансы и цены вводятся вручную.
    - Значит admin нужен раньше публичных API, иначе тебе нечем будет наполнять систему.
4. Сделать read-only API для контента.
    - /movies/, /movies/{id}/
    - /sessions/, /sessions/{id}/
    - /sessions/{id}/seats/
    - Это даст фронту стабильную основу до начала сложной логики брони.
5. После этого делать booking workflow.
    - POST /bookings/ для hold на 10 минут
    - POST /bookings/{id}/confirm/
    - DELETE /bookings/{id}/
    - Здесь уже нужны транзакции, защита от двойного бронирования и корректная работа expires_at.
6. Затем вынести механику статусов мест в отдельный сервисный слой.
    - Расчет available/held/booked
    - Очистка протухших SeatHold
    - Возврат expires_at и, желательно, server_time для фронтового таймера
7. В конце добить эксплуатационные вещи.
    - Docker-заготовки
    - README
    - env-конфиги
    - логирование бронирований
    - тесты на критические сценарии
