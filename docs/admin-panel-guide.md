# Работа с админкой

## Назначение

Этот файл описывает, как заполнять данные через Django Admin на текущем этапе проекта.

Админка нужна для ручного заведения:

- фильмов
- залов
- сеансов
- цен
- просмотра пользователей
- просмотра бронирований и hold-записей

## Как запустить

```bash
python manage.py migrate
python manage.py createsuperuser
python manage.py runserver
```

После запуска:

- открыть `http://127.0.0.1:8000/admin/`
- войти под superuser

## В каком порядке заполнять данные

Рекомендуемый порядок такой:

1. `Movies`
2. `Halls`
3. `Screenings / Movie sessions`
4. `Ticket prices`
5. `Users`, если нужен тестовый пользователь вручную
6. `Bookings` и `Seat holds` только для просмотра или точечной отладки

Причина простая:

- сеанс нельзя создать без фильма и зала
- цену нельзя создать без сеанса
- бронирование зависит от пользователя и сеанса

## 1. Movies

Сначала заполняй фильмы.

Поля:

- `title` — название фильма
- `description` — описание
- `genre` — жанр из списка
- `duration` — длительность в минутах
- `age_rating` — возрастной рейтинг
- `poster_url` — ссылка на постер
- `release_date` — дата релиза
- `is_active` — показывать фильм в афише или нет

Пример:

- `title`: `Dune: Part Two`
- `description`: `Epic science fiction film`
- `genre`: `sci_fi`
- `duration`: `166`
- `age_rating`: `12+`
- `poster_url`: `https://example.com/posters/dune-2.jpg`
- `release_date`: `2024-03-01`
- `is_active`: `true`

Рекомендации:

- `duration` всегда больше нуля
- для MVP удобно держать `is_active=true` только у актуальных фильмов
- `poster_url` можно оставить пустым, если фронтенд пока не требует постер

## 2. Halls

После фильмов заводи залы.

Поля:

- `name` — уникальное имя зала
- `rows` — количество рядов
- `seats_per_row` — мест в каждом ряду
- `schema_metadata` — JSON со схемой зала

Минимальный рабочий вариант:

- `name`: `Hall 1`
- `rows`: `8`
- `seats_per_row`: `12`
- `schema_metadata`:

```json
{
  "rows": [
    {
      "row": 1,
      "seats": [
        {"number": 1, "type": "standard"},
        {"number": 2, "type": "standard"},
        {"number": 3, "type": "vip"}
      ]
    }
  ],
  "disabled_seats": [
    {"row": 1, "number": 12}
  ]
}
```

Практический совет:

- если пока нет готовой детальной схемы, временно ставь `rows` и `seats_per_row`, а `schema_metadata` оставляй `{}` или минимальным JSON
- позже лучше привести `schema_metadata` в полное состояние, потому что выбор мест будет опираться именно на схему

## 3. Movie Sessions

Когда есть фильмы и залы, создавай сеансы.

Поля:

- `movie` — фильм
- `hall` — зал
- `start_datetime` — дата и время начала
- `end_datetime` — дата и время окончания
- `is_active` — активен ли сеанс

Пример:

- `movie`: `Dune: Part Two`
- `hall`: `Hall 1`
- `start_datetime`: `2026-04-28 19:00`
- `end_datetime`: `2026-04-28 21:46`
- `is_active`: `true`

Важно:

- `end_datetime` должен быть больше `start_datetime`
- в одном зале нельзя создать два сеанса на одинаковое `start_datetime`

В админке у сеанса уже есть inline для цен. Это удобный основной способ заполнения цен.

## 4. Ticket Prices

Цены лучше заполнять прямо внутри карточки сеанса через inline.

Поля:

- `session` — сеанс
- `amount` — цена
- `currency` — валюта
- `pricing_source` — источник цены

Рекомендуемый вариант для MVP:

- `currency`: `KGS`
- `pricing_source`: `manual`

Пример:

- `session`: `Dune: Part Two @ Hall 1`
- `amount`: `450.00`
- `currency`: `KGS`
- `pricing_source`: `manual`

Ограничение:

- для одного сеанса нельзя создать две цены с одной и той же валютой

Если нужен только один тариф на сеанс, создавай одну запись в `KGS`.

## 5. Users

Пользователей можно создавать:

- через API регистрации
- вручную в админке

Для тестов вручную через админку можно создать:

- обычного пользователя
- staff/superuser для доступа к админке

Рекомендация:

- обычных пользователей для бизнес-проверок лучше заводить через API регистрации
- через админку создавать только admin/staff или тестовые учетные записи

## 6. Bookings

`Booking` лучше не использовать как стартовую ручную сущность. Это уже результат сценария бронирования.

Поля:

- `session`
- `user`
- `total_amount`
- `booking_status`
- `payment_status`
- `confirmed_at`

Когда можно заполнять вручную:

- для отладки интерфейса admin
- для тестовых кейсов
- если нужно посмотреть, как будут выглядеть связанные `BookingSeat`

Минимальный пример:

- `session`: выбрать существующий сеанс
- `user`: выбрать существующего пользователя
- `total_amount`: `900.00`
- `booking_status`: `pending`
- `payment_status`: `pending`

Потом внутри booking через inline можно добавить места.

## 7. Booking Seats

`BookingSeat` отражает конкретные места внутри брони.

Поля:

- `booking`
- `seat_row`
- `seat_number`
- `price_at_booking`

Пример:

- `booking`: существующая бронь
- `seat_row`: `3`
- `seat_number`: `7`
- `price_at_booking`: `450.00`

Ограничение:

- внутри одной брони одно и то же место нельзя добавить дважды

## 8. Seat Holds

`SeatHold` тоже не является стартовой сущностью для обычного ручного заполнения. Это техническая таблица удержания мест.

Поля:

- `session`
- `user`
- `seat_row`
- `seat_number`
- `expires_at`
- `status`

Когда использовать вручную:

- для отладки будущей логики бронирования
- для теста отображения занятых и удерживаемых мест

Пример:

- `session`: существующий сеанс
- `user`: существующий пользователь
- `seat_row`: `4`
- `seat_number`: `9`
- `expires_at`: время через 10 минут от текущего
- `status`: `held`

Важно:

- для одного сеанса нельзя держать одно и то же место в статусе `held` дважды
- для одного сеанса нельзя держать одно и то же место в статусе `booked` дважды

## Что заполнять в первую очередь для MVP

Если нужно быстро получить рабочие данные для фронта и следующих этапов, достаточно такого минимума:

1. создать 3-5 фильмов
2. создать 2-3 зала
3. создать для каждого фильма несколько сеансов
4. добавить цену в `KGS` для каждого сеанса

Этого уже хватит для:

- афиши
- страницы фильма
- расписания
- перехода к будущему выбору мест

## Чего пока не делать вручную массово

Пока лучше не заполнять массово:

- `Booking`
- `BookingSeat`
- `SeatHold`

Причина:

- эти сущности должны в основном появляться из сценария бронирования, а не из ручного ввода

## Короткий рабочий сценарий администратора

1. Открыть `Movies` и создать фильмы.
2. Открыть `Halls` и создать залы.
3. Открыть `Movie sessions` и создать сеансы.
4. Внутри каждого сеанса сразу добавить цену.
5. Проверить, что фильмы и сеансы активны.
6. После этого можно отдавать данные фронтенду и переходить к следующему этапу.
